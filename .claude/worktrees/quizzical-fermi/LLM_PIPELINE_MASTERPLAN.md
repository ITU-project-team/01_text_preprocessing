# 🥕 당근마켓 UMC 연결성 분석: 완벽 통합 파이프라인 (LLM 최적화)

본 문서는 **수백만 건의 당근마켓 원본 데이터**에서 **UMC 6차원 분류 및 긍/부정 감성 분석**을 도출하기까지의 전체 워크플로우를 "LLM API 적용"이라는 대전제 하에 가장 효율적으로(비용 최적화 & 정확도 극대화) 구성한 마스터플랜입니다.

---

## 🎯 전체 파이프라인 개요

기존 형태소 분석 기반의 빡센 전처리 파이프라인을 버리고, **"문맥을 가장 잘 이해하는 LLM의 특성"과 "API 비용 최적화"**에 초점을 맞추어 재설계했습니다.

## 🛠️ Phase 상세 가이드

### Phase 0: 키워드 발견 및 확장 (Keyword Discovery)

기존에 구현 완료된 파이프라인으로, 사람이 선정한 넓은 의미의 '씨앗 키워드(Seed Keywords)'를 바탕으로 해당 게시글들을 무작위 추출한 뒤, LLM(gpt-4o-mini)을 이용해 **당근마켓 특유의 은어/축약어가 포함된 "정밀 키워드(Precise Keywords)"**를 자동 추출하여 `keywords.yaml`로 저장합니다.

- **산출물**: `config/keywords.yaml` (UMC 6차원별 상세 키워드 딕셔너리)

### Phase 1: 가벼운 정제(Light Cleaning) 및 키워드 필터링

**[기존 문제점]**
이전의 `normalize.py`는 Kiwi를 이용해 문법적인 조사, 어미를 전부 날려버리는 하드-전처리 방식이었습니다. 이는 LLM이 문맥 뉘앙스(특히 긍/부정)를 파악하는 데 오히려 방해가 됩니다.

**[개선된 로직]**

1. **가벼운 형태 정제 (비용 다이어트 및 검색 정확도 향상)**
   - `ㅋㅋ`, `ㅠㅠ`, `ㅎㅎ` 등 3회 이상 과도하게 반복되는 자모음을 최대 2자로 축약. (예: `ㅠㅠㅠㅠㅠㅠ` → `ㅠㅠ`)
   - 앞뒤의 불필요한 연속 공백 제거 및 줄바꿈 정리.
   - 띄어쓰기가 아예 없는 극단적인 글만 가볍게 형태소 기반 띄어쓰기 교정 적용. (키워드 매칭을 위해)
2. **키워드 필터링**
   - Phase 0에서 만들어진 정밀 키워드(`keywords.yaml`)가 포함된 게시글만 추출.

- **산출물**: `data/processed/01_filtered.csv`

### Phase 2: 차원 분류 및 감성 분석 (분기형 파이프라인)

`01_filtered.csv`의 잔여 데이터 개수와 예산에 따라 유연하게 전략을 선택합니다.

#### **Track A: 전수 LLM API 분석 (데이터 20만 건 이하 / 예산 허용 시)**

가장 정확하고, 가장 빠른 지름길입니다.

- 필터링된 모든 데이터를 병렬 비동기(Async) `gpt-4o-mini` API에 통과시킵니다.
- 프롬프트 설계 핵심: "너는 당근마켓 게시글에서 디지털 소외 계층의 문제(UMC)를 분석하는 전문가다. 다음 글의 UMC 차원(다중 가능)과 긍정/부정 상태를 JSON으로 추출해라."
- **예상 비용**: 10만 건당 약 1만 원 내외.
- **소요 시간**: Async 사용 시 10만 건 기준 1~2시간 이내 처리 완료.

#### **Track B: Teacher-Student 증류 (데이터 50만 건 이상 / 예산 극도 제한 시)**

- **Teacher (LLM)**: 5,000건만 무작위(층화) 샘플링하여 GPT API로 고품질 라벨 정답지 제작. (~$1 미만)
- **Student (SLM)**: 생성된 5,000건의 정답지로 `KcELECTRA` (또는 Llama-3 8B QLoRA)를 커스텀 파인튜닝.
- 파인튜닝된 로컬 모델을 돌려서 나머지 49만 5천 건을 무료로 분류.

### Phase 3: 정확도 검증 (Human Evaluation)

데이터 신뢰도를 입증하기 위해, 어떠한 트랙(Phase 2)을 타든 **반드시 수행해야 하는 필수 검증 과정**입니다.

1. 팀원 3명이 각각 무작위 추출된 게시글 300~500건을 엑셀로 열어 직접 UMC 차원 및 긍/부정을 라벨링합니다. (Cross-Validation).
2. 합의된 100% 완벽한 **Human Gold Standard** 정답지(`gold_test.csv`)를 생성합니다.
3. 우리가 돌린 LLM 스크립트(혹은 파인튜닝 모델)가 내놓은 답안과 `gold_test.csv`를 비교하여 F1-Score, Accuracy, 혼동 행렬(Confusion Matrix) 지표를 도출하고 보고서에 삽입합니다.

### Phase 4: 지역별 집계 및 지도 시각화

최종적으로 UMC 차원이 분류된 `[자치구, 차원, 긍/부정]` 정보를 `groupby`하여 집계합니다.

- 서울시 행정동/자치구 시군구 `geojson` 경계 데이터를 활용해 지리 데이터 프레임(`geopandas`)으로 변환합니다.
- `folium` 혹은 `deck.gl` 기반 라이브러리를 사용해 Choropleth(단계구분도) Map을 렌더링하고, UMC 차원별 디지털 격차가 높은 지역(Red Zone)을 시각화합니다.
- **산출물**: `notebooks/visualization.ipynb` 및 인터랙티브 `.html` 지도 파일.

## 💡 베테랑 리뷰 포인트 제언 (For PM & Senior Dev)

해당 마스터플랜에 대해 베테랑 시니어 그룹에서 다음 세 가지를 중점적으로 피드백 주시기를 요청합니다.

1. **Phase 1 (가벼운 전처리) 스코프**: LLM 입력단 앞단에서 "연속 자모음/공백 제거" 외에 띄어쓰기 교정을 어디까지 빡세게 할 것인가? (띄어쓰기를 너무 하면 원본 구어체 뉘앙스가 훼손될 우려 vs 안 하면 키워드 필터링 누락 우려)
2. **Phase 2 의사결정 기준점 (Threshold)**: Track A(전액 LLM 결제)와 Track B(파인튜닝)를 가르는 기준 건수를 20만 건(약 2.5만 원)으로 설정한 것에 대한 타당성.
3. **Phase 3 평가지표 합의 수준 (Cohen's Kappa)**: Human Gold Standard 생성 시 팀원 간 일치도(Inter-Annotator Agreement)를 어느 정도 선까지 용인할 것인가?
